## Identity

FORGE is an open-source, CLI-first agentic harness that orchestrates both local and frontier LLMs to build software. It is not a coding agent — it is the environment, constraints, feedback loops, and orchestration logic that turn models into a functioning engineering team. FORGE's unique contribution is **boundary measurement**: empirically tracking where local models suffice and where frontier judgment is required, per task type, over time. The dashboard headline is frontier/local delta, not task completion rate.

**Current state:** Phase 0 complete. Phase 1 implementation in progress. The v0.2 spec is locked. A scaffolded Python project exists with CLI skeleton, ACI tool interface definitions, Oracle pipeline stubs, config schema, benchmark cartridge template, and `feature_list.json` defining the Phase 1 work queue.

## Architecture

Full specification: **docs/FORGE_ARCHITECTURE_v0.2.md**

### Three Critical Design Axioms

1. **Fresh context is non-negotiable.** Workers get clean, bounded context per task. Long context (1M) is for planners only. This is an architectural axiom, not a tunable parameter. (Overview §11)
2. **The Oracle mediates all frontier interaction.** Frontier models never see the full codebase — they see a structured 2–4K token snapshot generated by a custom tree-sitter pipeline. This keeps frontier costs bounded and predictable. (Architecture §4)
3. **Enforcement is layered and mechanical.** L1 hooks run on every tool call (syntax, secrets, allowlist). L2 architectural linters run on every commit. L3a Desloppify mechanical runs continuously via tree-sitter. L3b Desloppify subjective runs at milestone boundaries via LLM. The model cannot override the enforcement layer. (Architecture §6)

## Code Standards

- **Python 3.11+** — minimum version, use modern syntax
- **Type hints required** on all public functions and methods
- **Tests required** for all new functionality — use `pytest`
- **Commit message format:** `[FORGE] <component>: <description>`
  ```
  [FORGE] oracle: Implement tree-sitter codemap extraction
  [FORGE] gate: Add single-reviewer verdict loop
  [FORGE] aci: Add search_file tool with 50-result bound
  ```
- **No hardcoded paths** — all paths come from `.forge/config.yaml` or are relative to project root
- **No hardcoded secrets** — API keys referenced via env var names in config (`api_key_env` field), never plaintext
- **Every ACI tool returns structured JSON** — not free-form text, not bare strings
- **All tool outputs are bounded** — max results, max lines, max tokens. No tool can flood the context window.
- **Error messages are agent-readable** — include WHAT is wrong, WHY it matters, and HOW to fix it
- **JSON schemas in the spec are normative** — match them exactly
- **ASCII diagrams in the spec are structural references** — use them to understand topology, not as implementation blueprints

## Decision Authority

### DECIDED items are locked

Every item marked **DECIDED** in `docs/FORGE_OVERVIEW_v0.2.md` and `docs/FORGE_ARCHITECTURE_v0.2.md` is a locked decision. These include but are not limited to:

- Boundary measurement is a core feature, not a metric
- Gate engine is a trust system (Phase 1: single reviewer + escalation)
- Desloppify is two architecturally distinct systems (mechanical + subjective)
- Skills crystallize downward (Prompt → YAML → Linter → Test → LoRA)
- Shadow mode before merge authority
- Error taxonomy and benchmark cartridges from day one
- Oracle is text-first, two-tier, custom tree-sitter (no RepoPrompt dependency)
- Fresh context is an axiom (workers get clean state per task)
- Grok is Intelligence, not a Reviewer
- DGX Sparks are independent (no linked topology)
- Human is sovereign (high visibility, low mandatory intervention)

**Do not reopen these decisions.** They may be revisited in v0.3+ based on empirical data, but they are not open for re-litigation during build.

### OPEN items

Items marked **OPEN** in the spec (Q11–Q18 in Overview §15) can be resolved through implementation. When you resolve an OPEN item:

1. Implement your chosen approach
2. Document the decision in a comment block at the top of the relevant module
3. Add a brief entry to `.forge/decisions/<question-id>.md` with: decision, rationale, alternatives considered

### Blocked items

If you are blocked — the spec is ambiguous, two spec sections conflict, or a Phase 1 feature requires a design decision not covered in the spec:

1. Create `.forge/blocked/<issue>.md`
2. Include: description of the block, options you've identified, your recommendation
3. Move to the next feature in `feature_list.json`
4. Do not guess. Do not invent spec. Do not proceed on assumptions that could require rework.

### Immutable rules

- **Never modify tests to make them pass.** Fix the code.
- **Never delete features from `feature_list.json`.** Mark as `passes: true` when complete.
- **Never modify `docs/FORGE_OVERVIEW_v0.2.md` or `docs/FORGE_ARCHITECTURE_v0.2.md`.** These are the spec — locked.
- **Never commit secrets, API keys, or credentials.**
- **Always commit with clean tests passing.** If tests fail, fix them before committing. If you exhaust context, roll back to the last clean commit.

## Session Protocol

Every work session follows this pattern:

```
1. Boot (steps 1–7 above)
2. Pick next incomplete feature from feature_list.json
3. Read the relevant spec section (referenced in feature_list.json or by component)
4. Implement the feature
5. Write tests for the feature
6. Run full test suite — all tests must pass
7. Commit with enforced format: [FORGE] <component>: <description>
8. Mark feature as passes: true in feature_list.json
9. If context is running low: commit clean state, stop
10. If blocked: create .forge/blocked/<issue>.md, move to next feature
11. Repeat from step 2
```

If you exhaust your context window: roll back to the last clean commit. The next session picks up from there. No partial state carries over — this is the fresh context axiom applied to the build process itself.
